# 网页端控制硬件执行器功能

## 功能概述

实现了网页端远程控制硬件执行器的功能。用户可以在网页上点击执行器的开关按钮，控制指令会通过服务器传递给硬件端执行。

---

## 架构设计

### 数据流程

```
网页端 → 服务器数据库 → 硬件端 → 执行设备
  ↓           ↓            ↓          ↓
发送指令   存储指令     查询指令    执行指令
  ↓           ↓            ↓          ↓
更新状态   确认完成     确认结果    反馈状态
```

### 技术方案

采用**轮询方式**：
1. 网页端发送控制指令到服务器
2. 服务器将指令存入数据库（状态：pending）
3. 硬件端定期查询服务器获取待执行指令
4. 硬件端执行指令并更新数据库（状态：executed）
5. 服务器更新执行器状态

---

## 数据库设计

### 控制指令表（actuator_commands）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 指令ID（主键） |
| actuator_id | VARCHAR(20) | 执行器ID |
| command | ENUM('on', 'off') | 控制命令 |
| status | ENUM('pending', 'executed', 'failed') | 指令状态 |
| created_at | TIMESTAMP | 创建时间 |
| executed_at | TIMESTAMP | 执行时间 |

## API接口

### 1. 发送控制指令（网页端）

**POST /api/actuators/[id]/commands**

请求示例：
```json
{
  "command": "on"
}
```

响应示例：
```json
{
  "success": true,
  "data": {
    "id": 1,
    "actuator_id": "WP-002",
    "command": "on",
    "status": "pending"
  },
  "message": "OK"
}
```

### 2. 查询控制指令（硬件端）

**GET /api/actuators/[id]/commands**

响应示例：
```json
{
  "success": true,
  "data": {
    "id": 1,
    "actuator_id": "WP-002",
    "command": "on",
    "status": "pending",
    "created_at": "2026-04-13T04:00:00.000Z"
  },
  "message": "OK"
}
```

### 3. 确认指令执行（硬件端）

**PATCH /api/actuators/[id]/commands**

请求示例：
```json
{
  "command_id": 1,
  "status": "executed"
}
```

响应示例：
```json
{
  "success": true,
  "message": "OK"
}
```

## 注意事项

1. **硬件端查询频率**：建议每1-2秒查询一次，避免过于频繁
2. **指令超时**：如果指令长时间未执行，可以手动清理数据库中的pending指令
3. **网络稳定性**：确保硬件端和服务器之间的网络连接稳定
4. **并发控制**：同一时间只能有一个待执行的指令
5. **状态同步优先级**：服务器状态优先于硬件状态，确保数据一致性

## 状态同步机制

### 概述

为确保服务器状态绝对权威性，实现了强制同步机制。**服务器状态为唯一基准**，硬件端必须无条件同步服务器状态。

### 核心原则

1. **服务器状态优先**：服务器状态是唯一权威状态
2. **硬件状态仅监控**：硬件上传的状态仅用于服务器监控，不会更新服务器执行器状态
3. **强制同步**：当硬件状态与服务器不一致时，硬件必须强制同步服务器状态
4. **无条件执行**：硬件端必须无条件执行服务器返回的状态

### 强制同步流程

```
硬件端上传状态
    ↓
服务器对比状态
    ↓
状态一致？ → 返回成功 → 硬件继续运行
    ↓
状态不一致？ → 返回服务器状态 → 硬件强制同步
    ↓
硬件执行同步
    ↓
重新上传确认
```

### API响应格式

#### 1. 强制同步响应

当硬件状态与服务器状态不一致时：

```json
{
  "success": true,
  "action": "force_sync",
  "data": {
    "server_state": "on",
    "server_mode": "auto",
    "hardware_state": "off",
    "hardware_mode": "auto",
    "mismatch": {
      "state": true,
      "mode": false
    }
  },
  "message": "状态不一致，必须强制同步服务器状态"
}
```

**硬件端必须执行的操作**：
1. 解析 `server_state` 和 `server_mode` 字段
2. **无条件执行**服务器状态，不管硬件当前是什么状态
3. 调整硬件执行器状态与服务器一致
4. 重新上传状态确认同步成功

#### 2. 同步成功响应

当硬件状态与服务器状态一致时：

```json
{
  "success": true,
  "action": "sync_success",
  "data": {
    "server_state": "on",
    "server_mode": "auto"
  },
  "message": "状态一致，同步成功"
}
```

**硬件端操作**：
- 状态一致，继续正常运行
- 无需额外操作

### 硬件端实现示例

```c
// 处理服务器响应
void HandleServerResponse(const char* response) {
    // 解析JSON响应
    cJSON* root = cJSON_Parse(response);
    cJSON* action = cJSON_GetObjectItem(root, "action");
    
    if (strcmp(action->valuestring, "force_sync") == 0) {
        // 状态冲突，必须强制同步服务器状态
        cJSON* data = cJSON_GetObjectItem(root, "data");
        const char* server_state = cJSON_GetObjectItem(data, "server_state")->valuestring;
        const char* server_mode = cJSON_GetObjectItem(data, "server_mode")->valuestring;
        
        printf("[强制同步] 服务器状态: %s, 服务器模式: %s\n", 
               server_state, server_mode);
        
        // 无条件执行服务器状态
        if (strcmp(server_state, "on") == 0) {
            TurnOnActuator();  // 强制开启
        } else if (strcmp(server_state, "off") == 0) {
            TurnOffActuator();  // 强制关闭
        }
        
        // 同步模式
        if (strcmp(server_mode, "auto") == 0) {
            SetAutoMode();
        } else if (strcmp(server_mode, "manual") == 0) {
            SetManualMode();
        }
        
        // 重新上传状态确认同步
        delay_ms(100);  // 等待硬件状态稳定
        UploadActuatorStatus();
        
    } else if (strcmp(action->valuestring, "sync_success") == 0) {
        // 状态一致，同步成功
        printf("[同步成功] 硬件状态与服务器一致\n");
    }
    
    cJSON_Delete(root);
}
```

### 典型场景示例

#### 场景1：网页端控制水泵

1. **用户操作**：在网页端点击"开启水泵"
2. **服务器状态**：`state: "on"`
3. **硬件状态**：`state: "off"`（硬件还未收到指令）
4. **硬件上传状态**：`{"state": "off"}`
5. **服务器响应**：
   ```json
   {
     "action": "force_sync",
     "data": {
       "server_state": "on",
       "hardware_state": "off"
     }
   }
   ```
6. **硬件执行**：强制开启水泵
7. **硬件确认**：重新上传 `{"state": "on"}`
8. **同步完成**

#### 场景2：硬件端误操作

1. **硬件状态**：`state: "on"`（硬件被误开启）
2. **服务器状态**：`state: "off"`（应该是关闭状态）
3. **硬件上传状态**：`{"state": "on"}`
4. **服务器响应**：
   ```json
   {
     "action": "force_sync",
     "data": {
       "server_state": "off",
       "hardware_state": "on"
     }
   }
   ```
5. **硬件执行**：强制关闭水泵
6. **同步完成**

### 状态同步优先级规则

1. **服务器状态绝对优先**：服务器状态是唯一权威状态
2. **硬件状态仅监控**：硬件上传的状态不会更新服务器状态
3. **强制执行**：硬件必须无条件执行服务器状态
4. **自动恢复**：硬件状态异常时自动恢复到服务器状态

### 状态同步时机

1. **硬件端定期上报**：每5秒上报一次状态
2. **指令执行后**：执行完指令后立即上报新状态
3. **强制同步后**：收到强制同步指令后立即执行并确认
4. **状态变化时**：检测到硬件状态变化时立即上报

### 数据库记录

所有硬件上传的状态都会记录到 `actuator_status_history` 表中，`trigger_source` 字段标记为 `hardware_report`，用于监控和调试。

### 优势

- ✅ **状态一致性保证**：服务器状态是唯一权威
- ✅ **自动纠错**：硬件状态异常时自动恢复
- ✅ **简化逻辑**：硬件端只需执行服务器状态
- ✅ **易于调试**：详细的状态对比日志
- ✅ **防止误操作**：硬件误操作会被自动纠正

## 故障排查

### 问题1：网页端发送指令失败

检查：
- 服务器是否运行
- 执行器ID是否正确
- 网络连接是否正常

### 问题2：硬件端查询不到指令

检查：
- 硬件端网络连接是否正常
- 执行器ID是否匹配
- 数据库中是否有pending状态的指令

### 问题3：指令执行后状态未更新

检查：
- 硬件端是否正确调用确认接口
- 数据库更新是否成功
- 前端是否自动刷新
